整体看，项目现在功能链路是健康的：go test ./...、npm test、npm run build 都通过。真正需要优化的点主要是“日志规模变大后的性能”和“前端复杂度继续膨胀后的维护成本”。

优先级最高

完整日志存储需要重构
现在 store.go (line 653) 的 AppendSessionTranscript 每次终端输出 chunk 都会：
读取整个 JSON -> 解密整段日志 -> 拼接 -> 重新加密 -> MarshalIndent -> 写回整个文件。
这在小日志没问题，但长时间 SSH 会话会越来越慢，甚至影响终端输出流畅度。建议下一版改成：

transcript 分块存储，例如 log_id + seq + encrypted_chunk
写入批量缓冲，100-300ms flush 一次
读取时按 chunk 流式拼接
同步时也更容易做增量同步
JSON 单文件存储已经到边界
store.go (line 821) 每个操作都读写整个 config.zen，而且 saveLocked (line 861) 使用 json.MarshalIndent。主机少、日志少时很好维护；但加完整日志、同步、SFTP 历史后，建议迁到 SQLite 或至少把日志拆成独立文件/目录。
我的倾向：主配置继续 JSON，日志改成独立加密 chunk 文件；后续同步压力最小。

终端内存缓存会无限增长
TerminalPane.jsx (line 119) 前端也会把每个 session 的输出字符串完整存在 buffersRef。长会话会占内存，而且切换 tab 会重写整段内容。建议限制前端 buffer，比如保留最近 1-5MB 或最近 N 行；完整内容以后以后端日志为准。

第二优先级
4. 前端状态中心过大
useAppState.js (line 13) 已经集中管理 Vault、Host、Session、SFTP、Log、新标签页、UI 状态。后面继续加同步/日志搜索/终端回放，会越来越难判断副作用。建议拆成：

useVaultState
useWorkspaceState
useSessionState
useHostState
useLogState
SFTP 组件过大
frontend/src/components/SftpWorkspace.jsx 现在约 1765 行，是项目里最大的前端文件。建议按“本地面板、远端面板、传输操作、路径栏、弹窗/菜单”拆开。这个不急，但越早拆，后面越不容易牵一发动全身。

xterm 尺寸逻辑依赖私有字段
TerminalPane.jsx (line 17) 和日志查看器都用了 terminal._core。这是解决错位问题的实用办法，但属于 xterm 内部 API，升级 xterm 时可能坏。建议封装成一个共享 terminalGeometry 工具，并给它补单测/降级逻辑。

第三优先级
7. 去掉生产 console
KeychainPanel.jsx (line 117) 有多处 console.log/error。桌面端调试还好，但凭据相关页面最好统一走 UI 错误提示或 Wails runtime log，避免生产环境输出敏感上下文。

缺少 lint/typecheck
前端目前只有 build/test，没有 ESLint/TypeScript。这个项目 UI 状态多，建议至少加 ESLint，先管住 hooks deps、未使用变量、console、复杂度。暂时不用强行迁 TypeScript，但后端模型和前端字段已经开始有大小写/蛇形转换，长期 TS 会有收益。

样式系统需要收敛 token
CSS 文件总量不小，sftp.css、session-log.css、app-shell.css 都在扩张。建议下一轮 UI 优化时补一层“工作区通用变量”：顶部栏高度、暗色终端栏、按钮状态、列表行高度、面板间距统一起来。你最近一直在调高度/颜色，本质就是这里缺统一约束。
